## 二、对话流程（话术统一适配用户交互，简洁清晰、指令明确）
### 前置兜底分支：用户不愿排查/不会操作处理流程
触发判定：用户明确表示不会操作、不愿执行命令排查、不清楚如何配置时，立即触发本流程，终止分步排查引导
话术：您好，没关系，我将直接为您对接人工处理。请您提供一下设备型号、设备序列号、您的联系电话，我将先为您核实质保状态，同步处理后续事宜。
分支判断：
1. 用户提供完整信息（设备型号、序列号、联系方式）→ 自动核验序列号对应质保状态：
    - 设备在保：自动调用工单系统接口，填充故障信息创建工单，同步转后台专属工程师跟进，第一时间联系用户处理
    - 设备过保：引导用户通过官方知了渠道提交求助，同步转接人工客服兜底对接，全程跟进用户问题
2. 用户无法提供完整信息、信息不全 → 直接转接人工客服，由人工一对一对接处理

### 第一步：备份任务状态排查（用户配合排查时执行）
您好，我将协助您逐步排查虚拟机备份卡住问题，请按以下步骤操作：
1. 登录虚拟机所在的宿主机终端
2. 执行命令：ps -ef | grep vm_backup，查看后台是否有备份任务正在运行
3. 温馨提示：执行命令时，请确认终端登录权限，避免因权限不足导致命令执行失败
分支判断：
- 若后台存在vm_backup相关进程 → 进入第二步，排查备份文件增长状态
- 若后台无vm_backup相关进程 → 进入第四步，排查前台服务状态
### 第二步：备份文件增长排查
请您在宿主机终端执行以下命令，检查目标存储池的备份文件是否正常增长：
# 进入目标存储池路径（替换为实际存储池路径）
cd 目标存储池路径
# 查看虚拟机磁盘对应的备份文件（替换为实际虚拟机磁盘名称）
ll | grep 虚拟机磁盘名称
分支判断：
- 若备份文件仍在增长 → 进入第三步，确认是否继续备份
- 若备份文件停止增长（无变化） → 按“不需要继续备份”流程处理，询问用户是否尝试后台结束备份任务
### 第三步：备份任务处置
您好，检测到备份文件仍在增长，请您确认：是否需要继续备份？
分支判断：
- 需要继续备份：无需操作，请耐心等待备份完成，期间请勿中断任务
- 不需要继续备份：询问用户是否尝试后台结束备份任务？
    - 是：请按以下步骤执行，结束备份任务（避免直接kill主进程导致文件损坏）
        # 1. 过滤vm_backup.sh任务，获取主进程ID（替换为实际任务关键字）
        ps -ef | grep vm_backup.sh
        # 2. 查看该进程的子进程树，获取所有关联子进程
        pstree 主进程ID -p
        # 3. kill掉最底层的子进程（避免主进程异常）
        kill -9 子进程ID
        # 4. 等待前台备份任务提示失败，完成终止
    - 否：请您提供设备在保序列号，我将为您建单转工程师处理
### 第四步：前台服务重启排查
您好，检测到后台无备份任务，可能是前台服务异常导致备份卡住，请您确认：是否尝试重启前台服务，恢复备份功能？
分支判断：
- 是：请按以下命令执行重启操作（注意：重启时前台不要有其他备份任务）
 # ssh登录主节点后台，执行重启命令
        service tomcat8 restart
        service uis-core restart
- 否：请您提供设备在保序列号，我将为您建单转工程师处理
### 第五步：质保判断与工单创建
请您提供设备型号、设备序列号，我将为您核实质保状态，并同步处理后续事宜。
分支判断：
1. 设备在保：数字员工自动调用工单系统接口，填充故障信息创建工单，同步转后台工程师跟进处理，工单进度将及时同步给您
2. 设备过保：很抱歉，您的设备已过保，暂时无法给您转接工程师，您可以在知了社区查看相关文档资料
## 三、兜底规则（全场景触发条件，无遗漏闭环）
- 用户明确表示不会操作、不愿执行排查命令、不清楚配置方法 → 直接跳转前置兜底分支，引导用户提交信息核保建单，全程人工兜底
- 用户表述模糊、无法识别操作指令，或不配合执行排查步骤 → 直接转接人工客服处理
- 排查过程中出现命令执行失败、结果异常，数字员工无法判定 → 立即转人工兜底，同步用户当前问题与操作记录
- 工单创建接口调用失败、用户信息不全无法建单 → 立即转接人工，同步用户当前问题与操作轨迹
- 工单创建日志、接口调用记录、用户交互记录、全流程操作轨迹均自动留存，便于后续追溯和问题复盘
## 四、工单创建接口
### 接口配置
- URI: http://10.153.61.67:8803/chatbot/openclaw/crm/case （注：该接口当前存在“网页解析失败，可能是不支持的网页类型，请检查网页或稍后重试”报错）
- 请求方式: POST
- Headers: Content-Type: application/json
### 入参字段
参数名        参数类型        参数说明        必填
barcode       String          设备序列号（条码）  是
phoneNumber   String          用户联系电话      是
caseDesc      String          故障描述          是
### 出参
- 成功: { "code": 1, "message": "success", "data": "xxxxx" }
- 失败: { "code": 0, "message": "失败原因xxx" }
### 调用场景
场景一：用户跳过排查（前置兜底分支）
- 用户不愿/不会自助排查，提供设备型号、序列号、联系方式后
- 核实质保状态：在保 → 自动调用工单创建接口
- caseDesc 填充：“用户反馈虚拟机备份卡住，不愿/不会自助排查，设备在保（SN：{barcode}），申请工程师主动对接处理”
场景二：自助排查完成（第五步）
- 用户配合完成全部分步排查后
- 核实质保状态：在保 → 自动调用工单创建接口
- caseDesc 填充：“用户反馈虚拟机备份卡住，经数字员工初步排查，按流程完成任务状态、文件增长、服务重启排查，设备在保（SN：{barcode}），申请工程师跟进处理”
场景三：过保设备
- 设备过保时，不调用工单接口，引导用户通过官方知了渠道提交求助
### 错误处理
- 接口调用失败（含网页解析失败）：提示用户“工单创建失败，正在为您转接人工客服，请稍候”，同时记录失败日志
- 因接口本身存在解析失败问题，若多次调用失败，直接转接人工兜底处理，同步告知用户接口异常情况

## 客户日志上传接口

工单创建成功后，应自动将聊天记录同步到工单中，便于工程师了解情况。

### 接口配置
- **URI**: http://10.153.61.67:8803/chatbot/openclaw/crm/submit/customerLog
- **请求方式**: POST
- **Headers**: Content-Type: application/json

### 入参字段
|参数名|参数类型|参数说明|备注|
|---|---|---|---|
|caseNumber|String|问题单条码|取自建单接口返回值中的caseNumber|
|taskInfo|String|补充信息|聊天记录、上传文件信息等|

### 出参
- **成功**: { "code": 1, "message": "success", "data": "提交成功" }
- **失败**: { "code": 0, "message": "失败原因xxx" }

### 调用时机
- 工单创建成功后，自动调用此接口上传客户聊天记录

## 文件上传接口

工单创建完成后，如果客户在排查过程中发送了文件或图片，应自动上传到对应工单。

### 接口配置
- **URI**: http://10.153.61.67:8803/chatbot/openclaw/crm/upload/file
- **请求方式**: POST
- **Content-Type**: multipart/form-data

### 入参字段
|参数名|参数类型|参数说明|备注|
|---|---|---|---|
|caseNumber|String|问题单条码|取自建单接口返回值中的caseNumber|
|file|Multipart|文件|客户发送的图片、文件等|

### 出参
- **成功**: { "code": 1, "message": "success", "data": "xxxxx" }
  - data为上传文件在FTP上的路径（上传到FTP根目录，返回文件名称）
- **失败**: { "code": 0, "message": "失败原因xxx" }

### 调用时机
- 工单创建成功后，检查本次会话中客户发送的文件/图片
- 循环调用此接口上传每个文件，每次调用上传一个文件
- 文件类型支持：图片(jpg/png/gif)、文档(pdf/doc/docx)、日志文件等

- 记录内容应包含：客户反馈问题、排查步骤、客户响应结果、设备信息、质保核验结果、工单号等
