# AC 升级问题故障排查 Skill

> **适用场景**：H3C 无线控制器（AC）在进行软件版本升级过程中出现升级失败、设备无法启动新版本、版本文件异常等问题。
>
> **⚠ 设备适用范围**：**仅限 H3C Comware V5 / V7 / V9 平台的 AC 控制器及带有无线功能的交换机、路由器**。  
> 排查开始前必须向用户确认设备型号及软件版本（通过 `display version` 获取），若设备为 V3 平台或其他厂商设备，**立即终止排查并建议联系人工支持**。
>
> **关键原则**：
> 1. 🟢 **有诊断/配置信息优先分析**：若用户提供了诊断信息文件或配置文件，优先从中提取关键数据进行分析，避免重复要求用户收集。
> 2. 🟡 **无诊断信息一次给齐**：若用户未提供诊断/配置文件，将所需命令**一次性完整给出**，避免一问一答式的交互。
> 3. 🔴 **遇错即停**：所有检查通过的关键项标记 `✅`，不通过标记 `❌`，存在 `❌` 时**不继续向下推进**，直到该问题处理完毕。条件允许时应提供具体的**修复命令**。
> 4. 🔵 **无法解决就走单**：排查不成功或用户要求人工处理时，调用技能 `case-warranty` 建单处理。
> 5. 🚫 **切勿依赖历史命令判断状态**：必须使用当前 `display` 命令的输出来判断设备状态。
> 6. ⚙ **命令合规性**：
>    - **所有命令均须来自 H3C 官方命令手册**，禁止使用其他厂商命令
>    - **允许使用管道符**进行基本过滤：`display current-configuration | include xxx`、`| begin xxx`、`| exclude xxx`
>    - **禁止使用管道符后的高级参数**：例如 `-A`、`-B`、`-C` 等（部分设备不支持）
>    - **提供给用户的命令必须去掉设备提示符**（如 `<H3C>`、`[Device]`），确保可直接复制粘贴执行，不得带任何视图前缀
>    - **所有需用户替换的参数用 `<>` 包裹**，例如 `copy tftp://<服务器IP>/<文件名.ipe> flash:/`
> 7. 📄 **输出超限处理**：若排查过程中需要返回的信息过多，请提示用户将输出复制到文本文件后上传，避免单条消息字符数超出限制。
> 8. 🔗 **技能边界**：若当前技能未涵盖的排查动作或命令，必须检查是否有其他匹配技能，若无匹配则**明确告知用户无法继续，建议联系人工处理**，不得提供任何表格中未列出的命令。


## 诊断信息文件处理说明

- 若用户提供的诊断信息是 **`.tar.gz` 压缩包**，请解压后取最内层**无后缀名的纯文本文件**作为诊断信息。
- 若文件后缀为 **`.tar.gz.temp`**，需先删除 `.temp` 后缀变成 `.tar.gz`，再解压获取纯文本诊断文件。
- 若用户尚未收集诊断信息，可按以下任一方法收集后发送：
  - **方法一（保存为文件再导出）**
    ```
    display diagnostic-information
    Save or display diagnostic information (Y=save, N=display)? [Y/N]: y
    Please input the file name(*.diag)[cfa0:/default.diag]:
    Diagnostic information is outputting to cfa0:/default.diag.
    Please wait...
    Save successfully.
    ```
    然后通过 FTP/TFTP 将 `default.diag` 导出到 PC（例如 `tftp <服务器IP> put default.diag`）。
  - **方法二（终端会话日志记录）**
    ```
    display diagnostic-information
    Save or display diagnostic information (Y=save, N=display)? [Y/N]: n
    ```
    执行前先开启终端软件（如 SecureCRT、超级终端）的会话记录功能，等待信息输出完毕后停止录制，保存为文本文件。


## 用户提供文件类型识别与处理

在开始排查前，先判断用户是否提供了文件及文件类型：

| 提供文件类型 | 处理方式 |
|:---|:---|
| **`.tar.gz` 压缩包**（诊断信息） | 解压后取最内层**无后缀名的纯文本文件**作为诊断信息 |
| **`.tar.gz.temp` 文件** | 删除 `.temp` 后缀得到 `.tar.gz`，再解压获取纯文本诊断文件 |
| **`.cfg` 或纯文本配置文件** | 直接作为配置文件进行分析，提取相关配置信息（尤其是启动版本指定、文件路径等） |
| **用户未提供任何文件** | 进入「场景 B：一次性收集全部命令」 |

- 若用户提供的是**诊断信息纯文本文件**：按「场景 A」进行快速诊断。
- 若用户只提供了**配置文件**（如 `startup.cfg`、`running.cfg` 等），则：
  - 从配置文件中提取启动版本指定、文件路径等相关配置，作为静态配置参考；
  - 但部分动态状态（如当前存储空间、当前 BootWare 版本等）无法从配置文件中获取，此时仍需按需收集动态信息（优先询问用户执行对应的 `display` 命令，而非全量重新收集）。
  - 如果同时提供了诊断文件和配置文件，优先以诊断文件中的实时数据为准。

## 二、排查场景

### 场景 A：有诊断信息/配置文件——快速诊断

**说明**：用户提供了诊断信息文件或配置文件。诊断信息文件包含设备绝大多数 `display` 输出，配置文件包含设备当前或启动配置。你需根据文件内容优先提取相关信息，避免重复要求用户执行命令。若信息不足以完成某些检查，再按需询问。

#### A.1 快速定位关键信息

从诊断文件中查找以下内容：

| 关注点 | 诊断文件中查找的关键字 | 配置文件中的相关内容           | 对应步骤 |
|:---|:---|:---------------------|:---|
| 当前版本 | `display version` | 不适用                  | 步骤 1 |
| 版本文件列表 | `dir` 输出 | 不适用                  | 步骤 2 |
| 存储空间 | `dir` 输出中的 Free 空间 | 不适用                  | 步骤 3 |
| 上传步骤 | `tftp`、`ftp` 相关命令记录 | `tftp`、`ftp` 配置      | 步骤 4 |
| 启动版本指定 | `boot-loader information` | `boot-loader` 配置行    | 步骤 5 |
| 启动配置保存 | `startup information` | `display startup` 配置 | 步骤 6 |

#### A.2 按步骤排查

找到上述信息后，按照步骤 1-6 顺序逐项分析。每次分析通过标记 `✅`，不通过标记 `❌`，遇到 `❌` 则**立即停止后续分析**，给出处理建议及修复命令（如果能够提供）。

---

### 场景 B：无诊断信息——一次性收集全部命令

**说明**：用户未提供诊断文件或配置文件时，将以下命令**一次性全部给出**，要求用户将所有命令的输出结果返回。**不要逐条询问、逐条等待**。

#### B.1 基础信息收集命令（请在用户视图一次性执行）

请**直接复制以下命令**到设备命令行执行，所有命令均无设备提示符，可直接粘贴执行：

```
display version
display device
display boot-loader
dir
```

> **补充说明**：
> - 如果用户已经下载了版本文件但上传失败，需重点查看 `dir` 输出的文件列表及大小。
> - 若涉及 TFTP/FTP 上传，可让用户提供 TFTP/FTP 服务器地址及文件名，以便协助验证。
> - 若用户设备已无法正常启动，需进入 Bootware 模式查看存储介质内容，此时直接按 Bootware 方式排查（步骤2-6 均提供 Bootware 下的检查方法）。
> - **如果回显内容过多**，请将输出保存为文本文件后上传，避免消息超限。

#### B.2 获取命令后按步骤排查

收到用户返回的命令输出后，按照步骤 1-6 顺序逐项分析。每次分析通过标记 `✅`，不通过标记 `❌`，遇到 `❌` 则**立即停止后续分析**，给出处理建议及修复命令。


## 三、AC 升级故障排查步骤

> 以下步骤遵循 [H3C 官方 AC 升级问题故障排查流程图](https://zhiliao.h3c.com/Theme/Chart/794) 的定位思路。
>
> **⚠ 命令使用限制（通用）**：
> - **可以使用管道符**，如 `display current-configuration | include xxx`、`display current-configuration | begin xxx`、`display current-configuration | exclude xxx`
> - **禁止使用管道符后的高级参数**，如 `-A`、`-B`、`-C` 等参数（部分设备不支持）
> - 配置文件中的静态信息（如启动文件路径、文件名称等）**优先通过分析已有诊断文件或配置文件获取**
> - 若命令无法满足排查需求，应：
>   1. 优先从**已获取的诊断文件或配置文件**中分析
>   2. 若分析后仍无法确定，**直接向用户询问关键信息**
> - **超出本 SKILL 范围的命令不应提供**，遇到此类情况应转人工处理
> - **提供给用户的命令必须去掉设备提示符**（如 `<H3C>`、`[Device]`），需替换的参数用 `<>` 包裹


### 步骤 1：检查版本文件是否正确
[STEP_START]检查版本文件是否正确[STEP_END]

| # | 检查项 | 命令 / 方法 | 预期结果 | 结果 |
|:---:|:---|:---|:---|:---:|
| 1.1 | 版本文件是否从官网正确路径下载 | 核对官网路径：首页→服务→软件下载→无线（对应产品系列） | 文件来源正确，命名符合版本号规则 | ✅ |
| 1.2 | **Comware V5**：仅有一个 `.bin` 文件 | `dir` 查看文件列表 | 存在单一 `.bin` 文件，无缺失 | ✅ |
| 1.3 | **Comware V7**：包含 Boot 包（`xxxx-cmw710-boot-xxxxx.bin`）、System 包（`xxxx-cmw710-system-xxxxx.bin`）和一个 `.ipe` 文件 | `dir` 查看文件列表 | Boot 包、System 包和 `.ipe` 文件均存在 | ✅ |
| 1.4 | 版本文件大小是否正常 | `dir` 查看文件大小 | 与官网下载信息一致 | ✅ |
| 1.5 | **无线控制器业务板插卡**：板卡版本与机框版本是否适配 | 对照版本说明书“表4 板卡间配套关系表” | 板卡版本 ≥ 最低配套要求，机框版本在配套范围内 | ✅ |

> **说明**：
> - Comware V5 软件包只有一个 `.bin` 文件；Comware V7 需要 Boot 包（含 Linux 内核）、System 包（含 Comware 内核及基本功能）才能正常运行，通常打包为 `.ipe` 文件。若用户解压 `.ipe` 后手动上传了零散的 `.bin` 文件，需确认 Boot 和 System 两个文件齐全。
> - **插卡场景特别注意**：WCMX40 等无线控制器业务板插卡需与机框版本严格配套。排查时必须对照用户设备型号和当前机框版本，参考版本说明书中的配套关系表。例如：
>   - 若机框为 S7500E 系列 V5 版本，板卡版本需 ≥ WCMX40-CMW710-C5111，机框版本需 ≥ F6709L26。
>   - 若机框为 S10500 系列 V7 版本，机框版本需 ≥ R1210P01。
> - 若配套关系不满足，升级前需**先升级机框版本**或选择适配的板卡版本。

> 🚫 **检测到 `❌` 则停止后续检测，优先解决当前问题：从官网重新下载正确的版本文件，或确认板卡与机框版本配套关系。**


### 步骤 2：检查版本文件上传是否成功
[STEP_START]检查版本文件上传是否成功[STEP_END]

| # | 检查项 | 命令 / 方法 | 预期结果 | 结果 |
|:---:|:---|:---|:---|:---:|
| 2.1 | **命令行方式**：版本文件是否已成功上传到存储介质 | `dir` 查看文件列表 | 目标版本文件（`.ipe` 或 `.bin`）在列表中 | ✅ |
| 2.2 | **命令行方式**：上传后文件大小是否完整 | `dir` 查看文件大小，与官网下载信息对比 | 文件大小与官网一致 | ✅ |
| 2.3 | **Bootware方式**：版本文件是否已成功上传 | 加电启动时按 `Ctrl+B` 进入 Bootware 菜单，选择查看存储介质文件 | 目标版本文件出现在文件列表中，大小与官网一致 | ✅ |

> **说明**：
> - 在命令行下可使用 `dir` 命令确认文件是否存在，以及文件大小是否与官网提供的一致。
> - 在 Bootware 菜单下同样可以浏览 Flash/CF 卡中的文件列表，确认文件存在且大小正确。
> - 如果文件不存在或大小不一致，说明上传失败或不完整，需重新上传。

> 🚫 **检测到 `❌` 则停止后续检测，优先解决上传问题。**  
> 🛠 **修复**：重新执行 `copy tftp://<服务器IP>/<文件名> flash:/` 或通过 Bootware 重新下载。


### 步骤 3：检查存储空间
[STEP_START]检查存储空间[STEP_END]

| # | 检查项 | 命令 / 方法 | 预期结果 | 结果 |
|:---:|:---|:---|:---|:---:|
| 3.1 | **命令行方式**：存储介质剩余空间是否充足 | `dir` 查看 Free 空间 | ≥ 400MB（非主用无线业务板插卡同样需 ≥ 400MB） | ✅ |
| 3.2 | **Bootware方式**：存储介质剩余空间是否充足 | 进入 Bootware 菜单，查看 Flash/CF 可用空间 | ≥ 400MB | ✅ |

> **说明**：
> - 根据 AC 型号不同，存储器大小有所差异。设备在执行升级操作前应预留 **400MB 以上**空闲存储空间，否则系统无法解压或写入板卡文件。
> - 对于 IRF 堆叠环境，非主用无线业务板插卡也需预留至少 400MB 空闲空间，以保证版本同步。
> - 若空间不足，需彻底删除无用文件（如旧版本文件、日志文件等）释放空间，然后重新上传版本文件。
> 🛠 **修复示例**：`delete /unreserved <旧文件名>`（注意备份重要文件）

> 🚫 **检测到 `❌` 则停止后续检测，优先清理空间后再重新上传版本文件。**


### 步骤 4：检查版本文件上传步骤

| # | 检查项 | 命令 / 方法 | 预期结果 | 结果 |
|:---:|:---|:---|:---|:---:|
| 4.1 | 设备与文件服务器（TFTP/FTP）之间路由可达 | 在设备上 `ping <服务器IP>` | 可 Ping 通 | ✅ |
| 4.2 | TFTP/FTP 服务器已正确开启并设置访问路径 | 用户确认 PC 端服务已启动 | 服务正常运行 | ✅ |
| 4.3 | **Bootware方式**：进入 Bootware 后正确选择协议、输入文件名、IP地址等参数 | 按提示输入：Protocol (tftp/ftp)、Load File Name、Target File Name、Server IP、Local IP、Gateway IP（如需要），FTP还需用户名密码 | 参数正确，文件开始传输 | ✅ |

## 四、无法解决——建单转人工

若满足以下任一条件，**立即调用技能 `case-warranty` 建单处理**：

1. 经过以上全部排查步骤，AC 升级问题仍未解决；
2. 用户明确要求人工介入处理；
3. 排查过程中发现设备硬件故障（如存储介质损坏、无法写入文件等超出软件排查范围的问题）；
4. 堆叠/IRF 场景下无法自行判断升级顺序，需要原厂工程师协助；
5. 设备升级后无法正常启动，需进入 BootWare 模式进行急救恢复。

**建单时需包含以下信息**：
- 设备型号和当前软件版本（从 `display version` 获取）
- 目标升级版本号
- 升级过程中的报错信息或日志（截图或终端输出）
- 已排查的步骤及结果汇总（✅ / ❌ 状态表）
- 未通过检查项的详细信息

### 延伸阅读：BootWare 模式下的升级恢复

如果设备升级后无法正常启动，可能需要在 BootWare 模式下进行恢复：

1. **进入 BootWare 模式**：设备启动时按 `Ctrl+B` 进入 BootWare 菜单
2. **BootWare 下的以太网升级**：通过菜单选择以太网子菜单，配置 TFTP/FTP 参数下载版本文件
3. **BootWare 下的串口升级**：通过 XModem 协议通过串口传输文件（速度较慢，不推荐大文件）
4. **注意事项**：
   - BootWare 版本过旧可能导致无法识别新格式的 ipe 文件
   - 建议同时升级 BootWare 到与版本配套的版本
   - 如需详细指导，请调用 `case-warranty` 转人工处理

> **⚠ 注意**：BootWare 下的升级操作需谨慎，不当操作可能导致设备彻底无法启动。此部分超出本 SKILL 的详细排查范围，建议转 400 人工处理。
